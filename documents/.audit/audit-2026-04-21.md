### billing-and-pricing.md — STALE

**Source:** https://developers.google.com/maps/billing-and-pricing/pricing
**Fetched:** 2026-04-21

**Stale claim #1: $200 monthly credit presented as the cornerstone of the pricing model**
- **Our doc says:** "Every Google Cloud account with Maps APIs enabled receives a $200 USD monthly credit applied automatically. For most small-to-medium applications, this credit covers all usage."
- **Our doc also says:** "## The $200 Monthly Credit — Applied automatically each month — no action required — Covers all Maps Platform APIs (Maps, Routes, Places) — Resets on the first of each month — Cannot be accumulated or rolled over — Applies per billing account, not per project"
- **Current Google page says (https://developers.google.com/maps/billing-and-pricing/pricing):** The page "does not mention a $200 monthly credit. There is no acknowledgment that such a credit was removed. The document contains no reference to any blanket monthly credit program." Pricing is organized around tiered volume discounts within Essentials/Pro/Enterprise SKU categories, and a Subscriptions program ("Subscribe to save!").
- **Corroboration (https://developers.google.com/maps/billing-and-pricing/march-2025 via web search):** "All Google Maps Platform users with billing enabled received $200 in recurring monthly credit until February 28, 2025, after which it was replaced by a free usage threshold." Essentials SKUs now get 10,000 free monthly events, Pro SKUs get 5,000, Enterprise SKUs get 1,000, Map Tiles get 100,000.
- **Recommendation:** Rewrite the "$200 Monthly Credit" section entirely. Replace with the per-SKU free-usage-threshold model (10k Essentials / 5k Pro / 1k Enterprise / 100k Map Tiles) and add a note that the flat $200 credit was retired Feb 28, 2025. Remove the "$200 credit covers approximately 28,500 Dynamic Maps loads..." back-of-envelope table — the math no longer applies.

**Stale claim #2: Routes API SKU names and price points**
- **Our doc says:** "| Routes: Compute Routes Basic | $5.00 | | Routes: Compute Routes Advanced | $10.00 | | Routes: Compute Routes Preferred (traffic) | $15.00 |"
- **Current Google page says:** "Routes: Compute Routes Essentials" at "$5.00" at baseline, reducing to "$0.38" at maximum tier. Google's current pricing is tiered by volume and uses the Essentials/Pro/Enterprise nomenclature — there is no "Compute Routes Basic/Advanced/Preferred" trio on the current page.
- **Recommendation:** Replace the Routes pricing table with the current Essentials tier plus a note that higher-traffic preferences roll up into Pro pricing. Indicate that prices decline with volume tiers.

**Stale claim #3: Autocomplete pricing as a flat per-1k price**
- **Our doc says:** "| Autocomplete (per session) | $2.83 | | Autocomplete without session token (per request) | $2.83 |"
- **Current Google page says:** "Autocomplete Requests (Essentials): $2.83 for 10,001-100,000 events, declining to $0.21 at highest volume."
- **Recommendation:** Note that $2.83 is the baseline/Essentials tier and that volume discounts apply. The current page no longer separates session vs. non-session pricing in that simple form.

**Stale claim #4: Dynamic Maps pricing shown as flat**
- **Our doc says:** "| Dynamic Maps | $7.00 |"
- **Current Google page says:** "Dynamic Maps (Essentials): Pricing ranges from $7.00 at the 0-100,000 tier down to $0.53 at 5,000,000+ monthly events."
- **Recommendation:** Mark $7.00 as the Essentials entry-tier price and reference the tiered discount schedule.

**Stale claim #5: Budget-alert guidance assumes the $200 credit**
- **Our doc says:** "Create a budget (e.g., $200/month to match the free credit)"
- **Current reality:** There is no "free credit" to match; free usage is now per-SKU thresholds.
- **Recommendation:** Drop the "to match the free credit" justification; recommend budgets calibrated to actual projected monthly spend.

### places-api.md — STALE

**Source:** https://developers.google.com/maps/documentation/places/web-service/overview
**Fetched:** 2026-04-21

**Stale claim #1: Legacy Places API described as still enable-able and mid-migration**
- **Our doc says:** "The Places API (New) is the current version of Google's place data service. It replaces the legacy Places API with improved field masking, session tokens, and richer place data."
- **Our doc migration table header says:** "## Migration from Legacy Places API — The legacy `PlacesService` is being replaced by the `Place` class"
- **Current Google page says:** "Places API (New) is the current version. Places API is now Legacy and can no longer be enabled."
- **Recommendation:** Change "is being replaced" to "has been replaced; legacy Places API can no longer be enabled for new projects" and surface that hard cutoff prominently at the top of the doc.

**Stale claim #2: Legacy `Autocomplete` widget shown as a current option alongside the new element**
- **Our doc says:** Under "## Autocomplete" it presents `new Autocomplete(document.getElementById("search-input"), {...})` as the primary example, then labels `PlaceAutocompleteElement` as "(New)" — implying both are live choices.
- **Current reality:** The legacy Places API (which the `google.maps.places.Autocomplete` widget is part of) "can no longer be enabled" per the overview page. The canonical element is `PlaceAutocompleteElement`.
- **Recommendation:** Reorder so `PlaceAutocompleteElement` is the primary example. Mark `google.maps.places.Autocomplete` as legacy and only retain it in a migration-from-legacy section.

**Stale claim #3: `AutocompleteService` + `AutocompleteSessionToken` example uses legacy class**
- **Our doc says:** "const service = new google.maps.places.AutocompleteService(); service.getPlacePredictions(...)"
- **Current reality:** `AutocompleteService` belongs to the legacy Places API that "can no longer be enabled." Session tokens in the new API are passed directly to `Place.fetchFields()` / `PlaceAutocompleteElement`.
- **Recommendation:** Replace the `AutocompleteService` session-token example with a `PlaceAutocompleteElement`-based session token example.

### deprecations-2026.md — STALE

**Source:** https://developers.google.com/maps/deprecations
**Fetched:** 2026-04-21

**Stale claim #1: Missing currently-listed deprecations**
- **Our doc says:** Covers `google.maps.Marker`, legacy `PlacesService`, Directions API → Routes API, inline map `styles` array, and Elevation API pricing. No mention of Drawing Library, Heatmap Layer, `addDomListener`, or Place Autocomplete `bounds/location/radius`.
- **Current Google page says:** Active deprecation list includes (verbatim items surfaced by the fetch): "Drawing Library (Aug 8, 2025) → Terra Draw", "Heatmap Layer (May 27, 2025) → deck.gl HeatmapLayer", "Maps JavaScript API addDomListener() (Apr 7, 2022) → addEventListener()", "Place Autocomplete bounds/location/radius (May 2023) → locationBias and locationRestriction", "google.maps.Marker (Feb 2024) → AdvancedMarkerElement".
- **Recommendation:** Add new sections for Drawing Library, Heatmap Layer, `addDomListener`, and Place Autocomplete bounds/location/radius. Each should cite the deprecation date Google lists.

**Stale claim #2: "Directions API → Routes API" framed as optional/recommended**
- **Our doc says:** "The legacy Directions API continues to work but the Routes API is the recommended replacement for server-side routing."
- **Current Google page says:** The deprecations page fetch did not surface a verbatim Directions API deprecation notice — but the Routes API overview's own migration guidance treats Directions and Distance Matrix as legacy APIs customers should transition *from*. Our doc's wording is softer than Google's current positioning.
- **Recommendation:** Tighten to "Directions API and Distance Matrix API are legacy; Google's current guidance is to migrate to the Routes API for both new and existing server-side routing." If you want a hard-deprecation quote, pull it from the Routes migration guide rather than the general deprecations page.

**Stale claim #3: Inline `styles` array shown as "deprecated" without current corroboration**
- **Our doc says:** "Legacy `styles` array in the Map constructor is deprecated."
- **Current reality:** The Google deprecations page fetch did not list "inline map styles" or the `styles` array as a named deprecation item. Cloud-based styling via `mapId` is strongly *recommended*, but the doc's use of the word "deprecated" is stronger than what the canonical deprecations list currently supports.
- **Recommendation:** Either downgrade the wording to "strongly discouraged / not recommended; Cloud-based styling via Map ID is the supported path" OR cite a specific Google source that calls the `styles` array deprecated. Otherwise the flag cannot be backed.

### troubleshooting.md — MIXED

**Fetched:** 2026-04-21

**Confirmed fresh:** `RefererNotAllowedMapError`, `ApiNotActivatedMapError`, `BillingNotEnabledMapError`, `InvalidKeyMapError` all appear verbatim in Google's current error codes list, and the recommended fixes (Cloud Console Credentials, enabling the API, linking a billing account, verifying the key) still align.

**Stale claim #1: Missing multiple currently-documented error codes**
- **Our doc says:** Lists four map-loading errors (RefererNotAllowedMapError, ApiNotActivatedMapError, BillingNotEnabledMapError, InvalidKeyMapError) plus JS-side ones.
- **Current Google page says (verbatim error list from fetch):** Also documents `ApiTargetBlockedMapError`, `DeletedApiProjectMapError`, `ClientBillingNotEnabledMapError`, `ExpiredKeyMapError`, `InvalidAppCheckTokenMapError`, `InvalidClientIdMapError`, `MalformedCredentialsMapError`, `MissingKeyMapError`, `ProjectDeniedMapError`, `RefererDeniedMapError`, `OverQuotaMapError`, `ApiProjectMapError`, `UrlAuthenticationCommonError`, plus warnings including `RetiredVersion`, `NoApiKeys`, `InvalidVersion`.
- **Recommendation:** Add at minimum `ExpiredKeyMapError`, `OverQuotaMapError`, `MalformedCredentialsMapError`, `InvalidAppCheckTokenMapError`, and `RetiredVersion` (warning). These are the ones RAG users will actually hit.

**Stale claim #2: CORS-error workaround recommends legacy `PlacesService` / `DirectionsService`**
- **Our doc says:** "Use the Maps JavaScript API's built-in services (`PlacesService`, `DirectionsService`)"
- **Current reality:** Per the Places API overview fetch, "Places API is now Legacy and can no longer be enabled." Recommending `PlacesService` as the fix directs readers to a class tied to a no-longer-enable-able legacy API.
- **Recommendation:** Replace the `PlacesService` recommendation with `Place` class / `PlaceAutocompleteElement`. `DirectionsService` is still acceptable for client-side routing per our own deprecations-2026.md (which notes it "remains the recommended approach for client-side route display"), so it can stay.

**Stale claim #3: OVER_QUERY_LIMIT retry example uses legacy callback shape**
- **Our doc says:** `const result = await geocoder.geocode({ address });` — and assumes `error.code === "OVER_QUERY_LIMIT"`.
- **Current reality:** Google's documented error code on the error-messages page is `OverQuotaMapError` for the JS API surface. `OVER_QUERY_LIMIT` is the web-service-side status string. The example conflates the two.
- **Recommendation:** Either convert the example to the web-service (HTTP) Geocoding API where `OVER_QUERY_LIMIT` is the real status string, or switch the check to the `OverQuotaMapError` code the JS API actually emits.
